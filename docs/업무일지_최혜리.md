## 2026년 4월 29일 (수) — Day 1 모델 학습 및 첫 캐글 제출

**한 것**
- GitHub 레포 생성 (`medicine-object-detection-project`)
- `.gitignore`, `requirements.txt` 작성 및 첫 커밋
- 구글 드라이브 데이터 업로드 (train/test/annotations 총 1.93GB)
- Colab Pro T4 GPU 환경 셋업, 드라이브 마운트
- COCO JSON 파싱 확인 → 이미지 232장, 어노테이션 763개, 카테고리 56개
- bbox 오버레이 시각화 → 정상 확인
- 클래스 불균형 시각화 → 최대 51배 (일양하이트린정 153개 vs 카나브정 3개)
- bbox 크기 분포 확인 → 평균 너비 250px, 높이 265px
- COCO → YOLO 포맷 변환 완료 (232개 라벨)
- `data.yaml` 생성 (클래스 56개)
- YOLOv8s 베이스라인 학습 (50epoch, imgsz=1024) → mAP50-95: **0.985**
- YOLOv8m 학습 완료 (50epoch) → mAP50-95: **0.990**
- YOLOv11s 학습 완료 (50epoch) → mAP50-95: **0.989**
- YOLOv11s 증강 버전 학습 → mAP50-95: **0.941**
- 캐글 첫 제출 → Public Score: **0.95788**

**인사이트**
- `imgsz=1024` 설정이 팀원 대비 점수 차이의 핵심 요인으로 추정
- train=val 설정이라 학습 점수와 캐글 점수 차이 있음 → 과적합 아님
- YOLOv8m이 s보다 수렴 속도 훨씬 빠름

**느낀 점**
- 모델 학습 중에는 다른 작업 병행하는 습관 생김
- 첫 제출에서 0.95 넘겨서 방향이 맞다는 확신이 생김

## 2026년 5월 2~3일 (토~일) — 파라미터 튜닝 집중

**한 것**
- 캐글 제출 10회 (wbf_d4 시리즈 전체)
- iou_thr / skip_box_thr 전수 튜닝 실험
  - skip=0.20 이하로 내려도 row 수 4160으로 동일 → 한계점 확인
  - iou_thr 0.45, 0.60 모두 0.97292로 0.55보다 낮음
- **iou=0.55, skip=0.20이 최적 파라미터로 확정**
- WBF equal weights(1.0, 1.0, 1.0) 조합 최종 확인 → **0.97414** 달성

**인사이트**
- skip_box_thr 한계점 존재 → 더 낮춰도 row 수 변화 없어 추가 개선 불가
- iou_thr은 0.55가 명확한 최적값

**느낀 점**
- "왜 안 되는지"를 체계적으로 확인한 게 더 중요한 성과임
- 파라미터 튜닝의 한계를 직접 경험함 → 다음 단계는 새 모델 시도

## 2026년 5월 4일 (월) — 새 모델 실험 및 한계 확인

**한 것**
- YOLOv11l 100epoch 학습 완료 → mAP50-95: 0.925
- v11l 예측 CSV 생성 및 앙상블 실험
  - v11l 단독 캐글 제출 → 0.74699 (과적합 확인)
  - v11l 4모델 앙상블 → 0.95524 (best보다 낮음)
- conf=0.15 재예측 3모델 앙상블 → 0.97414 (변화 없음)
- 외출 중 모델 3개 순차 학습 (약 6시간)
  - yolo11s_1280 (150epoch) → mAP50-95: 0.826
  - yolov8m_1280 (72epoch, 얼리스타핑) → mAP50-95: 0.724
  - yolo11s_aug_v2 (150epoch) → mAP50-95: 0.948
- 새 모델 예측 CSV 3개 생성 및 앙상블 실험 7종 캐글 제출
  - augv2 교체/추가 → 전부 0.97414 미만
  - v11s_1280 추가 → 0.91658 (대폭 하락)
  - skip=0.25 → 0.97414 동일

**인사이트**
- 모델 크기가 크다고 좋은 게 아님 → v11l이 v11s보다 캐글 점수 낮음 (과적합)
- 증강 강화(augv2)가 train mAP는 높아도 캐글 점수는 낮음 → 과증강은 일반화 성능 저하
- 1280 해상도 모델 추가 시 점수 하락 → 학습/테스트 해상도 불일치 가능성
- Colab 세션 유지: 브라우저 꺼지면 끊김 → 콘솔 자동클릭 코드 활용

**느낀 점**
- 6시간 연속 학습 돌려놓고 다른 일 하는 것처럼, AI 엔지니어 실무에서도 학습 중 병행 작업 습관이 생김
- 0.97414 벽이 단순 파라미터 조정으로는 절대 뚫리지 않는다는 걸 확인 → 구조적으로 다른 접근 필요

## 2026년 5월 5일 (화) — 새 모델 3개 학습 및 최고점 경신

**한 것**
- Soft-NMS 앙상블 실험 및 캐글 제출 → 0.96329 (WBF보다 낮음 → Soft-NMS 제외 확정)
- 외출 중 새 모델 3개 순차 학습 (총 약 2.5시간)
  - YOLOv8l (100epoch, imgsz=640) → mAP50-95: **0.991**
  - YOLOv9m (100epoch, imgsz=640) → mAP50-95: **0.990**
  - YOLOv11s 저lr (100epoch, lr=0.0005) → mAP50-95: **0.992**
- 새 모델 3개 예측 CSV 생성 (pred_v8l, pred_v9m, pred_v11s_lr)
- 앙상블 5종 조합 실험 및 캐글 제출
  - **wbf_4m_v8l → 0.97581 ✅ 최고점 경신** (기존 0.97414 돌파)
  - wbf_4m_v9m → 0.97374
  - wbf_4m_v11slr → 0.97300
  - wbf_6models → 0.97410
  - wbf_new3 → 0.96483
- 보고서 PDF 초안 완성 (8섹션)

**인사이트**
- WBF가 Soft-NMS보다 이 데이터셋에 확실히 적합 → 방식 최종 확정
- YOLOv8l(43M 파라미터) 추가로 0.97414 벽 돌파 → 구조적으로 다른 큰 모델이 앙상블 다양성에 기여
- 모델 수를 무작정 늘리면 오히려 낮아짐 → 6모델(0.97410) < 4모델(0.97581)
- 멘토 피드백 확인: 소규모 데이터셋에서의 과적합 위험, OOD 취약성 → 한계점으로 보고서에 반영

**느낀 점**
- 0.97414 벽이 파라미터 튜닝으로는 절대 안 뚫렸는데 새 모델 하나로 뚫린 게 인상적 → 결국 데이터/모델 다양성이 핵심
- 비개발자로 시작해서 앙상블, mAP, TTA, WBF까지 직접 실험하는 수준이 된 게 이번 프로젝트의 가장 큰 성과
- 멘토 피드백(과적합 가능성, OOD 취약성)이 날카롭지만 정확함 → 한계점을 솔직하게 보고서에 담은 것이 오히려 더 전문적
